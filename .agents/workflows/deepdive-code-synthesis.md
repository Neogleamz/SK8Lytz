### 🧠 Phase 2 — Deduplication and Synthesis

#### Step 1: Automated Exact-Match Dedup
Group all findings by `file:line` key. If two or more findings share the exact same `file` AND `line`:
- Keep the finding with the most detailed `description` and `suggested_fix`.
- Merge the `rule_violated` fields (a single line can violate multiple rules).
- Count how many agents flagged the same location (this is a **confidence signal** — more agents = higher confidence).

#### Step 2: Fuzzy Dedup (Same File, Adjacent Lines)
If two findings are in the same file within ±5 lines and cite the same `rule_violated`:
- Likely the same issue reported from slightly different perspectives.
- Merge into one finding with both line references.

#### Step 3: False Positive Scrubbing
Apply deep codebase knowledge to remove findings that are acceptable edge cases:
- `any` casts in test/mock files (`__tests__/*`, `__mocks__/*`) → acceptable, remove
- `Platform.OS` checks in files that are already platform-specific (e.g., `*.android.ts`) → false positive
- `try/catch` without `AppLogger.error` in `__DEV__`-gated diagnostic blocks → acceptable
- `setTimeout` in UI animation code (not BLE pipeline) → context-dependent, flag as LOW not remove
- Missing state matrix in pure utility components (no data fetching) → false positive

#### Step 4: Confidence Scoring
For each deduplicated finding, assign a confidence score:
- **🔴 CONFIRMED** — Flagged by 2+ independent agents (domain + sniper overlap)
- **🟡 PROBABLE** — Flagged by 1 agent with clear evidence (code snippet present)
- **⚪ UNCERTAIN** — Flagged by 1 agent, weak evidence, or high `false_positive_risk`

### 🌊 Phase 2.75 — Parallel Wave Assignment (AST-Verified — MANDATORY before Phase 3)

This phase prevents VS-014 (parallel worktree file collision). It MUST run before any task is written to the Bucket List.

#### Step 1: Write the Domain Clusters JSON
After completing Phase 2 Orthogonal Clustering, write the clusters to disk:
```
artifacts/domain_clusters.json  ← { "domain-name": [findings...], ... }
```

#### Step 2: Run the AST Collision Tool
```powershell
node tools/ast-parser.js --collision-matrix artifacts/domain_clusters.json
```
This tool:
- Walks the real TypeScript import tree (not filename overlap)
- Detects import-dependency collisions between domains
- Applies greedy graph coloring to assign `[WAVE:N]` tags
- Enforces the 8-task swarm cap automatically (Rule 10)
- Outputs JSON with `wave_assignments` and `waves`

**Manual collision calculation is FORBIDDEN.** Always use the tool output.

#### Step 3: Output the Batch Strategy Table
From the tool's `waves` output, write this table into `artifacts/system_audit_report.md` AND the Bucket List anchor block:
```
| Wave | Task Clusters | Parallel-Safe? | Prerequisite |
|------|--------------|---------------|-------------------|
| 1    | cluster-A    | Solo           | None              |
| 2    | cluster-B, cluster-C | ✅ Yes | Wave 1 merged |
```

#### Step 4: Assign `[WAVE:N]` Tags
Use the tool's `wave_assignments` map. Every cluster gets a `[WAVE:N]` tag before being written to the Bucket List. No cluster may be written without a wave assignment from the tool.

   MANDATORY ENFORCEMENT CONTRACT — NO EXCEPTIONS:
   [S1] Before writing anything, confirm you are NOT on master:
        git branch --show-current
        If output is 'master' → HALT. Do not proceed.

   [COMMIT] Stage and commit the failing test:
        git add __tests__/<domain>/<slug>.test.ts

### 📋 Phase 3 — Triage Routing

#### Step 1: Generate the Audit Report
Append to `artifacts/system_audit_report.md` (after the metrics header):
- Full list of unique verified findings, grouped by rule, sorted by severity (HIGH → LOW)
- Each finding includes: file, line, rule, severity, confidence, description, suggested fix
- Separate section for KNOWN_ISSUES matches (documented but not yet fixed)

#### Step 2: Draft Triage Batch (Orthogonal Domain Clustering)
**CRITICAL:** Do NOT group tasks by Rule (e.g. all R-08 violations). This causes cross-domain collisions. 
Instead, group findings into logical **Orthogonal Task Clusters** by Domain/File Sector (e.g. `[BATCH:sweep-src-context]`, `[BATCH:sweep-src-components-ui]`). 
- One cluster contains ALL rule violations (R-08, R-16, R-01, etc.) that occur within its specific file boundaries.
- This guarantees zero file overlap between clusters, maximizing parallel execution safety.

For each task cluster, draft a task summary with:
- **Goal:** "Fix the bug to make the auto-generated failing regression test pass."
- **Source of Truth:** `[Test File]` generated in Phase 2.85
- Affected files list (REQUIRED — used by Phase 2.75 collision check)
- Rule violations addressed
- Estimated size tag (`[Snack]` for <5 files, `[Meal]` for 5-15, `[Feast]` for 15+)
- Risk tag based on domain (`[H-RISK]` for BLE/protocol, `[M-RISK]` for UI, `[L-RISK]` for utils/docs)
- **`[WAVE:N]` tag** — assigned from Phase 2.75 graph coloring. REQUIRED on all synthesized tasks.
- **`Prerequisite: Wave N-1 fully merged`** — added to Details for any Wave N > 1.

#### Step 3: The Bucket List Guardrail
**⛔ Do NOT append tasks directly to the Bucket List.** You must first:
1. Generate `PLAN-*.md` files in `docs/plans/` for each task cluster.
2. Complete Phase 2.75 — every task MUST have a `[WAVE:N]` tag before it can be appended.
3. Write the **Batch Strategy Table** into the Bucket List anchor block (see Rule 8 of Kanban Constitution).
4. Route all findings through the `/intake` workflow OR present the full audit report to the user for manual triage.
5. Only tasks with approved plans, verified SoT fields, AND `[WAVE:N]` tags may enter the Triage Queue.

> ⚠️ **Do not stop after generating the report.** The synthesis workflow is complete only when:
> - All PLAN files are written to `docs/plans/`
> - All tasks are in the Bucket List with `[WAVE:N]` tags
> - The Batch Strategy Table is written to the anchor block
> Stopping at "report generated" is an incomplete execution.

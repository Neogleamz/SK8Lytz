# SK8Lytz App: System Master Reference (EXHAUSTIVE)

This is the definitive technical manual for the SK8Lytz LED control system. It consolidates all protocol specifications, pattern listings, hardware configurations, and implementation details necessary for development. **DO NOT SHORTEN THIS DOCUMENT.**

## 1. System Overview & Architecture
The SK8Lytz ecosystem provides high-performance, professional-grade LED control for roller skates.
- **Platform**: React Native (Expo)
- **Communication**: Bluetooth Low Energy (BLE) via `react-native-ble-plx`
- **Protocol**: Zengge / Magic Home / LEDnetWF (V2 Packet Wrapper)
- **Visualizer**: High-fidelity React Native `Canvas` / `View` based LED simulation.

## 2. Hardware Specifications

### Device Profiles

| Profile | LED Count | IC Type | Default Color Order | Usage |
|:---|:---|:---|:---|:---|
| **HALOZ** | 16 | WS2812B | GRB (`0x03`) | Wheel / Frame Ring |
| **SOULZ** | 43 | WS2812B | GRB (`0x03`) | Boot / Sole Strip |

### Hardware Configuration Command (`0x81`)
Executed upon connection to ensure software-to-hardware pixel alignment.
- **Hex Header**: `0x81`
- **Payload Structure**: `[0x81, PointsHigh, PointsLow, ColorOrder, IcType, 0x00, 0x00, Checksum]`
- **Color Order Key**: `RGB=1, RBG=2, GRB=3, GBR=4, BRG=5, BGR=6`
- **Example (HALOZ GRB)**: `[0x81, 0x00, 0x10, 0x03, 0x03, 0x00, 0x00, 0x97]`

## 3. Protocol Deep-Dive (Zengge V2)

### A. The V2 Packet Wrapper
All raw commands must be wrapped in this 8-byte header before transmission.

| Byte | Field | Description |
|:---|:---|:---|
| 0 | Version | `0x00` (Version 0) |
| 1 | Sequence | Incremental ID (`messageCounter++ % 256`) |
| 2-3 | Flags | `0x80 0x00` (No segmentation) |
| 4-5 | Total Length | Raw Payload Length (Big Endian) |
| 6 | Cmd Length | `RawPayloadLength + 1` |
| 7 | Cmd Family | `0x0B` (Standard Control) |
| 8+ | **Payload** | The raw command bytes. |

### B. Checksum Calculation
Applies to almost all raw payloads.
- **Logic**: Sum all bytes in the raw payload and apply `& 0xFF` (modulo 256).

## 4. Command Library (Hex Level)

### Power & Connectivity
- **Power ON**: `[0x71, 0x23, 0x0F, 0xA3]`
- **Power OFF**: `[0x71, 0x24, 0x0F, 0xA4]`
- **Discovery**: Zengge devices typically broadcast as `LEDnetXXXXXX`.

### Color & Brightness (`0x31`)
- **Format**: `[0x31, R, G, B, 0x00, 0x00, 0x0F, Checksum]`
- **Range**: `0x00` to `0xFF` (0-255).

### RBM (Running Business Mode) (`0x42`)
Modern high-performance pattern selection.
- **Format**: `[0x42, PatternID, Speed, Brightness, Checksum]`
- **Pattern IDs**: 1 - 100+ (See Master Pattern List).
- **Default SK8Lytz Values**: Brightness 90% (`0x5A`), Speed 50% (`0x32`).

### Multi-Color Segmented (`0x59`)
Individual LED control.
- **Format**: `[0x59, TotalLen(2), Colors(3N), NumPoints(2), Type, Speed, Direction, Checksum]`
- **Application**: 10 Fixed Modes (Comet, Dashed, etc.).

### Custom DIY Patterns (`0x51`)
Unlimited creativity with up to 32 steps.
- **Format**: `[0x51, Step1(8), Step2(8)... Step32(8), 0x0F, Checksum]`
- **Step Payload**: `[Active(0xF0/0x0F), TransitionMode, Speed, R1, G1, B1, R2, G2, B2]`
- **Transition Modes**: `0x3A` (Jump), `0x3B` (Gradual), `0x3C` (Strobe).

## 10. App Microphone Live Sync (`0x74`)

Used when `IsDeviceMic` is `0x00` in the `0x73` configuration.

- **Command**: `[0x74, Magnitude, Checksum]`
- **Magnitude**: `0x00` - `0xFF` (0-255) based on audio amplitude.
- **Workflow**: 
  1. Send `0x73` with `IsDeviceMic=0`.
  2. Stream `0x74` every 50-100ms.
  3. Send `0x73` with `IsDeviceMic=1` to return to hardware mic.

## 11. Advanced Reverse Engineering Observations

The following details were extracted from `com.zengge.blev2` (Surplife) string resources and reflect advanced system behaviors.

### Broadcast BLE Grouping Protocol
Surplife uses a specialized "Broadcast" system for synchronized multi-device control.
- **Data Table**: `broadcast_blue_network` (Columns: `id`, `network_id`).
- **Logic**: Individual devices are assigned to a `broadcast_blue_network` to receive synchronous commands.
- **Key Commands identified**:
  - `createBroadcastBleGroupIfNotExist`
  - `sendBroadcastDeviceGroupCommand`
  - `loopSyncBroadcastBleDeviceGroup`
- **Splicing Logic**: Symphony controllers support a "Splicing" setting to treat multiple physical strips as one continuous virtual strip. If splicing settings are inconsistent within a group, the app triggers a reset.

### Group Leadership
Large groups utilize a "Group Leader" logic to coordinate timing and pattern synchronization.
- **Error Condition**: `Detects that the current group does not have a group leader.` (Indicates a failure in mesh/broadcast coordination).

## 12. BLE Discovery & Identification (SPI/Symphony)

To ensure only compatible addressable controllers are shown to the user, the app uses a multi-layered filtering strategy during the BLE scan.

### A. Primary Identification (Manufacturer Data)
The most reliable way to identify a Zengge SPI/Symphony controller is by inspecting the **Manufacturer Data** field in the advertisement packet.

- **Offset**: Byte index 9.
- **Symphony Code**: `0x33` (51) or `0xBF` (191).
- **Logic**: If byte 9 is `0x33` or `0xBF`, the device is a Symphony-compatible addressable strip.

### B. Secondary Identification (Name Prefixes)
If manufacturer data is unavailable or the device is during first-time pairing, the following case-insensitive name prefixes are used:

| Prefix | Description |
|:---|:---|
| `LEDnet` | Standard default Zengge factory name. |
| `SK8` | Custom SK8Lytz branding. |
| `ZG` | Zengge reference designs. |
| `HALO` | SK8Lytz Halo variant. |
| `SOUL` | SK8Lytz Soul variant. |

### C. Service UUID Validation
All compatible controllers SHOULD advertise the primary Service UUID:
- `0000ffff-0000-1000-8000-00805f9b34fb`.

# SK8Lytz App: System Master Reference (EXHAUSTIVE)

This is the definitive technical manual for the SK8Lytz LED control system. It consolidates all protocol specifications, pattern listings, hardware configurations, and implementation details necessary for development. **DO NOT SHORTEN THIS DOCUMENT.**

---

## 1. System Overview & Architecture
The SK8Lytz ecosystem provides high-performance, professional-grade LED control for roller skates.
- **Platform**: React Native (Expo)
- **Communication**: Bluetooth Low Energy (BLE) via `react-native-ble-plx`
- **Protocol**: Zengge / Magic Home / LEDnetWF (V2 Packet Wrapper)
- **Visualizer**: High-fidelity React Native `Canvas` / `View` based LED simulation.

## 2. Hardware Specifications

### Device Profiles

| Profile | LED Count | IC Type | Default Color Order | Usage |
|:---|:---|:---|:---|:---|
| **HALOZ** | 16 | WS2812B | GRB (`0x03`) | Wheel / Frame Ring |
| **SOULZ** | 43 | WS2812B | GRB (`0x03`) | Boot / Sole Strip |

### Hardware Configuration Command (`0x81`)
Executed upon connection to ensure software-to-hardware pixel alignment.
- **Hex Header**: `0x81`
- **Payload Structure**: `[0x81, PointsHigh, PointsLow, ColorOrder, IcType, 0x00, 0x00, Checksum]`
- **Color Order Key**: `RGB=1, RBG=2, GRB=3, GBR=4, BRG=5, BGR=6`
- **Example (HALOZ GRB)**: `[0x81, 0x00, 0x10, 0x03, 0x03, 0x00, 0x00, 0x97]`

## 3. Protocol Deep-Dive (Zengge V2)

### A. The V2 Packet Wrapper
All raw commands must be wrapped in this 8-byte header before transmission.

| Byte | Field | Description |
|:---|:---|:---|
| 0 | Version | `0x00` (Version 0) |
| 1 | Sequence | Incremental ID (`messageCounter++ % 256`) |
| 2-3 | Flags | `0x80 0x00` (No segmentation) |
| 4-5 | Total Length | Raw Payload Length (Big Endian) |
| 6 | Cmd Length | `RawPayloadLength + 1` |
| 7 | Cmd Family | `0x0B` (Standard Control) |
| 8+ | **Payload** | The raw command bytes. |

### B. Checksum Calculation
Applies to almost all raw payloads.
- **Logic**: Sum all bytes in the raw payload and apply `& 0xFF` (modulo 256).

## 4. Command Library (Hex Level)

### Power & Connectivity
- **Power ON**: `[0x71, 0x23, 0x0F, 0xA3]`
- **Power OFF**: `[0x71, 0x24, 0x0F, 0xA4]`
- **Discovery**: Zengge devices typically broadcast as `LEDnetXXXXXX`.
-   Direction Boolean: Byte [138] // (0x01 Forward, 0x00 Reverse)
   Terminator: 0x0F
   Checksum: sum(payload) & 0xFF

=== ALTERNATIVE NON-ADDRESSABLE RGB PATTERNS (Node.js RAW DISCOVERIES) ===
When pushing 0x42 packets, certain devices running generic non-IC LED firmware exhibit alternative mappings. Documenting these overrides here per user request for future legacy controller compatibility layers:
- ID 56: Seven-Color Cross-Fade (Conflicts with our Symphony "Red Marquee")
- ID 63: White Gradual Change
- ID 64: Red/Green Cross-Fade
- ID 67: Seven-Color Strobe Flash
- ID 69: Green Strobe Flash
- ID 76: Seven-Color Jumping Change
* NOTE: The SK8Lytz system primarily targets Haloz / Soulz operating on WS2812B/SM16703 IC arrays, hence why these standard base-emitter mappings do not align mechanically with our SPI hardware.

### Color & Brightness (`0x31`)
- **Format**: `[0x31, R, G, B, 0x00, 0x00, 0x0F, Checksum]`
- **Range**: `0x00` to `0xFF` (0-255).

### RBM (Running Business Mode) (`0x42`)
Modern high-performance pattern selection.
- **Format**: `[0x42, PatternID, Speed, Brightness, Checksum]`
- **Pattern IDs**: 1 - 100+ (See Master Pattern List).
- **Default SK8Lytz Values**: Brightness 90% (`0x5A`), Speed 50% (`0x32`).

### Addressable Mode (`0x59`) - Multi-Color Array / DIY Multi-Color
- **Format**: `[0x59, SegmentLength, TransitionType, Speed, 0x01, R1,G1,B1, R2,G2,B2...]`
- *Note on DIY Multi-Color Parity*: The Zengge APK's advanced "Multi-Color" tab natively utilizes this exact `0x59` chunked buffer to build DIY animations. The UI labels map strictly to these `TransitionType` bytes:
  - `0x00`: **Static** (Color blocks sit still)
  - `0x01`: **Gradual** (Flowing/Breathing transition)
  - `0x02`: **Strobe** (Flashing segments)
  - `0x03`: **Running Water** (Hard jumping Marquee)
- **SegmentLength (Byte 1)** defines the size of the looping chunk. (Must be >=10 points to avoid hardware glitching).
- **Speed (Byte 3)**: `0x01` (Slowest) to `0x64` (Fastest, usually bounded `0x20`+).
- **Direction Boolean (Byte 138 - if length padded)**: `0x01` Forward flow, `0x00` Reverse flow.
- A single 1-pixel padded string to `0x59` `Type 1` safely replicates Solid Mode without `0x31` flickering glitches.

### Custom DIY Patterns (`0x51`)
Unlimited creativity with up to 32 steps.
- **Format**: `[0x51, Step1(8), Step2(8)... Step32(8), 0x0F, Checksum]`
- **Step Payload**: `[Active(0xF0/0x0F), TransitionMode, Speed, R1, G1, B1, R2, G2, B2]`
- **Transition Modes**: `0x3A` (Jump), `0x3B` (Gradual), `0x3C` (Strobe).

## 10. App Microphone Live Sync (`0x74`)

Used when `IsDeviceMic` is `0x00` in the `0x73` configuration.

- **Command**: `[0x74, Magnitude, Checksum]`
- **Magnitude**: `0x00` - `0xFF` (0-255) based on audio amplitude.
- **Workflow**: 
  1. Send `0x73` with `IsDeviceMic=0`.
  2. Stream `0x74` every 50-100ms.
  3. Send `0x73` with `IsDeviceMic=1` to return to hardware mic.

## 11. Advanced Reverse Engineering Observations

The following details were extracted from `com.zengge.blev2` (Surplife) string resources and reflect advanced system behaviors.

### Broadcast BLE Grouping Protocol
Surplife uses a specialized "Broadcast" system for synchronized multi-device control.
- **Data Table**: `broadcast_blue_network` (Columns: `id`, `network_id`).
- **Logic**: Individual devices are assigned to a `broadcast_blue_network` to receive synchronous commands.
- **Key Commands identified**:
  - `createBroadcastBleGroupIfNotExist`
  - `sendBroadcastDeviceGroupCommand`
  - `loopSyncBroadcastBleDeviceGroup`
- **Splicing Logic**: Symphony controllers support a "Splicing" setting to treat multiple physical strips as one continuous virtual strip. If splicing settings are inconsistent within a group, the app triggers a reset.

### Group Leadership
Large groups utilize a "Group Leader" logic to coordinate timing and pattern synchronization.
- **Error Condition**: `Detects that the current group does not have a group leader.` (Indicates a failure in mesh/broadcast coordination).

## 12. BLE Discovery & Identification (SPI/Symphony)

To ensure only compatible addressable controllers are shown to the user, the app uses a multi-layered filtering strategy during the BLE scan.

### A. Primary Identification (Manufacturer Data)
The most reliable way to identify a Zengge SPI/Symphony controller is by inspecting the **Manufacturer Data** field in the advertisement packet.

- **Offset**: Byte index 9.
- **Symphony Code**: `0x33` (51) or `0xBF` (191).
- **Logic**: If byte 9 is `0x33` or `0xBF`, the device is a Symphony-compatible addressable strip.

### B. Secondary Identification (Name Prefixes)
If manufacturer data is unavailable or the device is during first-time pairing, the following case-insensitive name prefixes are used:

| Prefix | Description |
|:---|:---|
| `LEDnet` | Standard default Zengge factory name. |
| `SK8` | Custom SK8Lytz branding. |
| `ZG` | Zengge reference designs. |
| `HALO` | SK8Lytz Halo variant. |
| `SOUL` | SK8Lytz Soul variant. |

### C. Service UUID Validation
All compatible controllers SHOULD advertise the primary Service UUID:
- `0000ffff-0000-1000-8000-00805f9b34fb`.

## 13. Fixed Mode (Pattern 1-10) Implementation

The "Fixed Mode" in the SK8Lytz app utilizes three distinct protocols to provide high-quality static and moving patterns on addressable hardware:

### 1. Standard Solid (Pattern 1)
- **Commands**: `0x59` (Segmented Multi-Color Repeating). Note: `0x51` produces 299-byte payloads crashing standard MTUs and freezing the app UI stack. To dynamically accommodate any LED point size without modulo remainders and avoid standard 0-speed hardware rejection, we force `NumPoints=1` mapped continuously with `Speed>0`.
- **Behavior**: Sets all pixels to a single color.
- **Reactivity**: Immediately updates on color picker or hue slider change.

### 2. Symphony Multi-Color (Patterns 2-5, 9-10)
- **Command**: `0x59`
- **Logic**: Sends an array of colors that repeats or shifts based on the controller's internal logic.
- **Pattern 2 (Single Dot)**: `[fg, bg, bg, bg, bg, bg, bg, bg]` (Repeating).
- **Pattern 3 (Comet)**: `[fg, 50%fg, 20%fg, bg, bg, bg]` (Creates a fading tail effect).
- **Pattern 4 (Dashed)**: `[fg, fg, fg, fg, bg, bg, bg, bg]` (4-pixel segments).
- **Pattern 5 (Alternating)**: `[fg, fg, bg, bg]` (2-pixel segments).
- **Pattern 9 (Wave)**: `[bg, bg, fg, fg, bg, bg]` (A moving 6-pixel wave).
- **Pattern 10 (Pinch)**: `[fg, bg, bg, bg, bg, fg]` (Inward/Outward movement from edges).

### 3. Custom Step Mode (Patterns 6-8)
- **Command**: `0x51`
- **Logic**: Defines a sequence of colors and a transition mode (Jump/Fade).
- **Pattern 6 (Breath)**: Fade between `fg` and `bg`.
- **Pattern 7 (Flash)**: Jump between `fg` and `bg`.
- **Pattern 8 (Strobe)**: Fast jump between `fg` and `bg` with fixed speed.

## 15. Zengge APK Reverse Engineering Findings (com.zengge.blev2)

### A. App Architecture & Payload Construction
1. **Hybrid Structure**: The Zengge app uses Flutter for all UI and business logic, while the native Android code acts as a BLE bridge.
2. **Payload Origin**: The actual hex arrays mapping to commands like `0x42`, `0x51`, `0x59` are constructed inside the compiled AOT Dart code (`flutter_assets`), not native Java.
3. **Native Bridge (`FlutterBleControl.java`)**: The Flutter layer passes complete byte payloads and configuration targets directly to native BLE write functions, meaning the native app simply transmits what Flutter computes.

### B. Device State Mapping (`DeviceState2SymphonyLamp`)
The native code tracks the state of connecting hardware using the following byte structure (tracked in a 14-byte `DeviceState` array):
- **Red/Green/Blue**: Mapped to state positions `j`, `k`, `l`.
- **HSV Settings (Symphony Lamps)**: 
  - `Hue`: Stored natively as variable `t`.
  - `Saturation`: Variable `w` (0-100).
  - `Brightness/Value`: Variable `x` (0-100).
- **Mode `0x97` (Static Color)**: Natively mapped from HSV settings to the RGB arrays during dynamic translations. When setting solid Symphony colors, the app defaults to mode `97` (decimal).

### C. Device Controller Types (Native Offsets)
- `Ctrl_Mini_RGB_Symphony_0xa1.java`: Handles 0xA1 control protocol types.
- `Ctrl_RGBCW_Both_0x25.java`: RGBCW devices use the identifier `37` (0x25) and map brightness combinations between RGB diodes and White/Warm diodes to emulate exact CCT temperatures and luminosities natively.

### D. Hardware IC Types & Discovery
- Identified natively supported IC types inside the settings mappings (`SymphonyICTypeItem`):
  1. `DIM` (1)
  2. `CCT` (2)
  3. `RGB` (3)
  4. `RGB & W` (4)
  5. `RGB & CCT` (5)
  6. `RGB / W` (6)
  7. `RGB / CCT` (7)
These mappings closely relate to the `setHardwareConfig` (`0x81`) command values utilized in the `ZenggeProtocol.ts`.

### CRITICAL HARDWARE FINDINGS (0x59 & Ambient Optics)
1. **Array Length Glitching**: Continuously transmitting a 0x59 payload with an array length of < 10 causes the first chunk of physical LEDs to glitch and flicker severely. The array length MUST be dynamically padded to fall >= 10 to establish a perfectly solid, stable loop on the controller.
2. **Matrix Transition State Locks**: The Symphony controller firmware natively ties `0x59` movement rules to the Transition Byte.
    - **Type `0x00` (Static Flow)**: Instructs the array to cascade and naturally loop across the skateboard (used for Marquee, Comet, and Alternating).
    - **Type `0x01` (Gradual)**: Systematically locks the array in place, freezing all internal gradients into an unmoving stream. This is required to prevent flickering on `Solid` pattern outputs.
3. **Camera Optic Saturation Scaling**: Raw environment telemetry gathered from device cameras contains heavily desaturated hex codes due to fluorescent glare. To map properly to addressable strips, the hex matrix must be mathematically normalized by completely subtracting the noise floor (minimum RGB channel) and instantly scaling the dominant channel to `255`, ensuring pure structural saturation reaches the BLE pipeline.

## 6. Advanced V5 Firmware Discoveries (LedNetWF)
*Discovered via HomeAssistant / LEDNetWF repository analysis against V5 microcontrollers.*

### Candle Mode (Flicker Simulation)
- The V5 firmware supports a native "Candle" effect that uses internal algorithms to realistically simulate a flickering flame.
- **Parameters**: It allows specific adjustments for baseline brightness, flicker speed, and base RGB lighting color to establish warm ambient atmospheres completely independently of the static `0x42` pattern bounds.

### Hardware Alarms & Timers
- **Offline Schedules**: V5 controllers possess dedicated memory banks to store daily and weekly schedule masks (alarms) independently of the mobile app.
- **Power & Action Triggers**: Hex arguments natively support `alarm-on` (Boot device), `alarm-off` (Kill power), `alarm-rgb` (Boot on specific Static Hex array), and `alarm-effect` (Boot into a specific Pattern ID).
- **Known Firmware Glitch**: While writing alarms to device memory typically succeeds and populates correctly inside the official Zengge visual interfaces, the execution of "Repeat Day Masks" contains firmware-level race conditions that occasionally prevent the alarms from physically firing at the target time. OFF schedules possess definitively higher reliability than ON/Effect schedules.

### Smear Payload Protocol (Raw Streaming Vectors)
- The raw pixel-by-pixel color buffer stream utilized by certain V5 apps is classified as a "Smear Packet".
- **Structure**: These are functionally massive variants of the `0x59` matrix mapped into 144-byte streams (48 pixels * 3 channels), containing dedicated bit flags for directional stream math (`1`=static, `2`=stream, `3`=strobe, `4`=jump).

## THE  x81 HARDWARE BOUNDARY PACKET
Zengge BLE controllers do *not* require the controlling smartphone application to manually scramble RGB sequence bytes to match the target LEDs (e.g., swapping bytes to GRB natively in JS before Bluetooth transmission).

Instead, the ecosystem relies on the **0x81 Configuration Command**. This command acts as a physical 'gateway setting' written directly into the controller's memory.**0x81 Format (SET Hardware Settings)**:
The packet format requires exactly 6 parameters prefixed by the 0x81 header flag.
NOTE: This structure changed based on deep protocol audits. Segments are definitively a single byte.
[0x81, ICByte, OrderByte, PointsHigh, PointsLow, SegmentsByte, Checksum]

- **ICByte**: Maps explicitly to the silicon driver type.
  - WS2812B:  x01
  - SM16703:  x02
  - SM16704:  x03
  - WS2811:  x04
  - SK6812:  x05
- **OrderByte**: Maps to the physical copper wiring layout of the LED diodes.
  - RGB:  x00
  - GRB:  x01 (Standard WS2812B)
  - BRG:  x02

## THE 0x63 / 0x0A HARDWARE RESPONSE PACKET (GET)
When querying the controller (e.g. via 0x10), the response arrives with the full Zengge Header.
The payload data starts at Index 7.

- **Index 7**: Command Header (Usually 0x0A or 0x63 or 0x81)
- **Index 8**: IC Type (`ICByte`)
- **Index 9**: Color Sorting (`OrderByte`)
- **Index 10**: Points High Byte
- **Index 11**: Points Low Byte
- **Index 12**: Segments Byte

## 0x59 PAYLOAD DESERIALIZATION CAVEATS
Because the hardware controller automatically unscrambles standard RGB packets based on the aforementioned  x81 config, all mobile app arrays (like  x59 running marquee multi-colors) MUST ALWAYS BE BROADCAST IN PURE R, G, B FORMAT. 

If the React Native visualizer renders 'green' when 'red' is selected, this is strictly a mathematical index parsing error inside the UI code, *not* a requirement to artificially scramble the Bluetooth transmission payload.

- **0x59 Structure Constraint**: The payload dynamic color string ALWAYS begins explicitly at **Index 3**.
  - [0x59, LengthHigh, LengthLow, R1, G1, B1, R2, G2, B2...]
- Reading visualizer chunks manually offset to Index 5 strictly induces a complete color channel phase shift (Reading B1 as R2, R2 as G2, etc). Always interpret colors starting directly after the LengthLow byte.

## BLUETOOTH LOW ENERGY (BLE) ATTENUATION SCALING
The Sk8Lytz Dashboard actively measures physical Bluetooth Signal Attenuation (RSSI) expressed in negative decibel-milliwatts (dBm) natively translating to Wifi-style visual meters (`wifi-strength-X`).

- **Excellent Quality (> -60 dBm)**: Direct Line-of-Sight, Minimal Interference. Packet streaming (0x59 Arrays / 0x31 Solid Matrix) will be completely structurally intact without frame shedding.
  - UI Visualization: `Green` | `wifi-strength-4`

- **Fair Quality (-61 dBm to -80 dBm)**: Distanced Connection / Minor Physical Occlusion (i.e. Body Blocking). Array streaming will occasionally delay response by 50ms-150ms.
  - UI Visualization: `Orange` | `wifi-strength-2`

- **Poor Quality (< -80 dBm)**: Extreme Attenuation / High Interference. Native physical controller packet drops will occur gracefully causing intermittent visually skipping frames or missing firmware overrides.
  - UI Visualization: `Red` | `wifi-strength-1`

- **Unknown / Booting (0 or Null)**: Connecting phase resolving physical connection states.
  - UI Visualization: `Grey` | `wifi-strength-outline`

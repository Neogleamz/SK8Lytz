### Design Decisions & Rationale

We are fully transitioning to a catalog-driven hardware architecture. By removing hardcoded 'HALOZ' and 'SOULZ' types and replacing them with properties derived purely from `LOCAL_PRODUCT_CATALOG`, we ensure the app can support arbitrary future hardware (like RAILZ) seamlessly without requiring codebase updates for basic rendering or configuration capabilities.

## Proposed Changes

### `src/components/DockedController.tsx`

- **[MODIFY]** Change `type ProductType = 'HALOZ' | 'SOULZ' | 'RAILZ';` to `type ProductType = string;` to fully support dynamic hardware strings fetched from the Supabase registry.

### `src/components/ProductVisualizer.tsx`

- **[MODIFY]** Clean up `type?: 'HALOZ' | 'SOULZ' | 'RAILZ' | string;` references, reducing them directly to `string` primitives. This will fix the compilation references dynamically generated by strict types in external configurations.

### `src/screens/Onboarding/HardwareSetupWizardScreen.tsx`

- **[MODIFY]** Remove the `cfg.type === 'SOULZ'` gate preventing custom point configs. Replace it with `getLocalProfileById(cfg.type)?.hardwareAllowsCustomPoints`, enabling RAILZ to establish custom LED points during the FTUE securely.

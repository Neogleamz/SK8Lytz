# Implementation Plan: fix/protocol-seq-counter

> **Risk:** H-RISK | **Size:** Meal | **Rule:** R-21  
> **Source Analysis:** [system_audit_report.md — CLUSTER 3](file:///C:/Users/Magma/.gemini/antigravity/brain/a2899729-4d77-4e6c-8f8c-d23919eb2b74/system_audit_report.md)

## Goal

Fix three confirmed protocol-core defects:
1. **Sequence counter split-ownership** — `BleWriteDispatcher.ts:244` generates a random `seqByte` for `0x40` chunk framing, while `ZenggeProtocol.ts:117-120` maintains its own `messageCounter` used by `wrapCommand`. Two independent counters = the controller sees out-of-sequence frames on chunked payloads.
2. **Hardcoded 54-pixel MAX_PIXELS cap** — `ZenggeProtocol.ts:656` caps `setMultiColor` at 54 pixels regardless of device HW config. Per Protocol Bible §0x59 (line 378), `numLeds_hi/lo` should reflect the physical strip length, not the pixel array length.
3. **TransitionType mapping correctness** — `useProtocolBuilder.ts:14-21` correctly maps 0x01–0x06 (no 0x00). Audit flagged this as a contradiction, but code matches Protocol Bible §0x59 lines 366-373. This is a **false positive** — the code is already correct. Plan step is a verification-only pass.

## Steps

### Step 1: Fix sequence counter ownership — single atomic owner
**What:** The sequence counter for `0x40` chunked framing must be owned by the `ZenggeProtocol` instance, not randomly generated by `BleWriteDispatcher`.

**Evidence:**
- `BleWriteDispatcher.ts:244` — `const seqByte = Math.floor(Math.random() * 256) & 0xFF;` — random, not sequential.  
  Source: [BleWriteDispatcher.ts:244](file:///C:/Neogleamz/AG_SK8Lytz_App/SK8Lytz/src/services/BleWriteDispatcher.ts#L244)
- `ZenggeProtocol.ts:117-120` — `getSequenceCounter()` properly increments `messageCounter mod 256`.  
  Source: [ZenggeProtocol.ts:117-120](file:///C:/Neogleamz/AG_SK8Lytz_App/SK8Lytz/src/protocols/ZenggeProtocol.ts#L117-L120)
- Protocol Bible §0x51: `seqNum increments per save operation (0x04, 0x05, 0x06...)`.  
  Source: [ZENGGE_PROTOCOL_BIBLE.md:228](file:///C:/Neogleamz/AG_SK8Lytz_App/SK8Lytz/docs/ZENGGE_PROTOCOL_BIBLE.md#L228)

**Changes:**
1. In `ZenggeProtocol.ts:117`, change `getSequenceCounter()` from `private` to `public` so the dispatcher can call it on the shared instance.
2. In `BleWriteDispatcher.ts:244`, replace `Math.floor(Math.random() * 256) & 0xFF` with `ZenggeProtocol.getNextChunkSeqByte()` — a new static method that delegates to `_instance.getSequenceCounter()`.
3. Add static method `getNextChunkSeqByte()` to the ZenggeProtocol class that returns the singleton's next sequence number. This keeps the counter atomic across all code paths.

**Verify:** Unit test confirms two sequential calls to `getNextChunkSeqByte()` return incrementing values (not random). Manual BLE log inspection: chunked 0x51 writes show monotonically increasing seqByte.

### Step 2: Remove hardcoded 54-pixel cap in `setMultiColor`
**What:** `MAX_PIXELS = 54` is an MTU-derived constant that should NOT exist in the protocol layer. The protocol builds the full payload; the transport layer (BleWriteDispatcher) handles chunking if the result exceeds MTU.

**Evidence:**
- `ZenggeProtocol.ts:656` — `const MAX_PIXELS = 54;` (hardcoded).  
  Source: [ZenggeProtocol.ts:656](file:///C:/Neogleamz/AG_SK8Lytz_App/SK8Lytz/src/protocols/ZenggeProtocol.ts#L656)
- `ZenggeProtocol.ts:659` — `const numPoints = Math.max(12, Math.min(MAX_PIXELS, colors.length));` — silently truncates >54 pixel arrays.
- Protocol Bible §0x59 line 378: `numLEDs_hi and numLEDs_lo represent the physical length of the hardware strip... If clamped to MAX_PIXELS = 54... transition effects will only animate on the first 54 LEDs`.  
  Source: [ZENGGE_PROTOCOL_BIBLE.md:378](file:///C:/Neogleamz/AG_SK8Lytz_App/SK8Lytz/docs/ZENGGE_PROTOCOL_BIBLE.md#L378)
- `HW_CONSTRAINTS.maxPoints = 300` already exists at `ZenggeProtocol.ts:70`.  
  Source: [ZenggeProtocol.ts:70](file:///C:/Neogleamz/AG_SK8Lytz_App/SK8Lytz/src/protocols/ZenggeProtocol.ts#L70)

**Changes:**
1. Replace `const MAX_PIXELS = 54;` with `const MAX_PIXELS = HW_CONSTRAINTS.maxPoints;` (300).
2. Update comment on L653-655 to explain: pixel cap is now set to hardware max (300). The transport layer (BleWriteDispatcher + `buildChunkedFrames`) handles MTU fragmentation.
3. Keep the `Math.max(12, ...)` floor — Protocol Bible §0x59 line 377 mandates minimum 12 pixels.

**Verify:** `setMultiColor` called with 150 pixels produces a payload with `numPoints = 150`, not 54. TSC passes. Existing tests still pass (they send ≤54 pixels, so behavior is identical for them).
